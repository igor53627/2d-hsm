# Agent Gateway provisioning channel — attested keystore install + in-TEE scope-id provenance (TASK-25 / G3)

The production deployment path for the Agent Gateway keystore. Today **no production
agent-gateway image exists** — every build is a debug build that loads a keystore from a host file
via the `lab-agent-keystore-from-file` feature, and `enclave_scope_id` is a TEST-ONLY sentinel
(`[0xe1;32]`). This doc specifies the **G3** gap carved out of TASK-18 (per the 2026-06-19 locked
maintainer decision Q6): a production release image, an attested AF_VSOCK + SNP channel that
installs the sealed keystore into the enclave, and an in-TEE-minted random `enclave_scope_id` so
the 18-2 scope-binding byte-compare is not security theater.

**Status: design (25-1, Understand phase).** Q1-Q8 below are the locked maintainer decisions
(2026-06-19). This doc is the input to a Full Matrix on the design before any implementation slice
(25-2 onward). It is NOT a wire-format spec yet — §N refers forward to the 25-2 protocol slice that
freezes the byte-exact framing.

**Scope: agent keystore ONLY.** The producer ML-DSA keystore (long-term Block Producer key, the
0x19 vsock surface, TASK-1.x / TASK-5) is a **separate task with its own lifecycle and threat
model** (see §6). G3 installs one `pq-agent-keystore-v1` body and mints one `enclave_scope_id`.

## Decisions (locked, 2026-06-19 — Q1-Q8)

| # | Decision | Rationale one-liner |
|---|---|---|
| **Q1** | **Mutual attestation.** Enclave → provisioner: SNP report (measurement + VCEK chain + nonce, verified via TASK-1.2 `snp_verify.rs`). Provisioner → enclave: operator-signed boot-authorization binding (measurement/TCB hash + `fleet_scope_id` + anchor config), verified vs an operator CA pinned in the enclave binary. Operator-CA model = quorum, **MVP = 1 higher-tier key** (mirrors the existing recovery-authority tier). Channel-binding (the report `report_data` nonce is tied to the vsock session) is **MUST**, not residual — a quote replayed on a different session must fail. | A single direction is exploitable: provisioner-only-verify ⇒ MITM substitutes config after attestation; enclave-only-verify ⇒ provisioner cannot know it is handing sensitive config to a genuine enclave. The TASK-7.7 anchor handshake is already mutual; reuse that pattern. The 18-2 compact's "underspecified and partly inverted" finding is resolved: the standard SNP direction is **provisioner = verifier, enclave = attestor**, and *separately* the enclave verifies the provisioner's signature. |
| **Q2** | **Enclave seals its own keystore.** The provisioner sends only a **config map** (public settings + public trust roots, Q4); the enclave mints `enclave_scope_id` in-TEE, constructs the `KeystoreBody` from scratch (genesis-equivalent initial state), and seals it under its measurement-derived key. The host persists the returned sealed blob; subsequent boots unseal that blob via the existing path. | Alternative (a) "provisioner ships a pre-built sealed blob" requires the provisioner to construct the keystore — including `enclave_scope_id` — which (i) leaks secp256k1 secret scalars + faucet state outside the enclave, and (ii) lets the provisioner select the scope id (defeating Q4). Under (b) the keystore plaintext never leaves the enclave; the host sees only a sealed blob. Reuses the existing `seal_body()` infra; the mint op is `genesis_body()` minus the sentinel. |
| **Q3** | **Mint once at provisioning, seal, never re-mint.** `enclave_scope_id` is generated by `getrandom` exactly once, at first provisioning, written into the sealed config, and read back at every subsequent boot. **Restore = the one explicit exception** (TASK-24 / TASK-25 AC#6): a restored keystore carries a NEW id, so caps minted before the backup fail the 18-2 compare. | Re-mint on every boot would invalidate every previously-minted cap (each carries the old id in its signed `scope_identity`) — one restart = wedge all operations until the operator re-issues the cap stream. Stability of the id across restarts is the invariant 18-2 rests on. |
| **Q4** | **Three-basket authority split + structural enforcement.** (A) Enclave-minted, host-uncontrollable: `enclave_scope_id` (only). (B) Provisioner-supplied over the attested channel: the 8 public fields `twod_chain_id, environment_identifier, admin_authority_pk, recovery_authority_pk, backup_recovery_wrapping_pubkey, fleet_scope_id, anchor_root, monotonic_treasury_config_version (=1 init)`. (C) Enclave-init deterministic: `authority_epoch = 0`. **Enforcement is structural**: the wire-format provisioner-config map has NO field for `enclave_scope_id` — the host cannot supply it because the protocol does not carry it. This is strictly stronger than a runtime "reject if host-supplied" check. | 18-2's whole adversary is a host that clones an enclave; if the host can pick or copy the id, the byte-compare is security theater. Structural impossibility > runtime reject (the latter can be forgotten or bypassed by a bug). The residual — a compromised operator selecting a malicious `admin_authority_pk` — is operator-trust by design (authorities are external trust roots, not enclave-enforced), and `fleet_scope_id` collusion across clones is closed by the handler-level 18-5 `financial ⇒ scope_class==0` policy, not by G3. |
| **Q5** | **Separate one-shot bootstrap vsock port, distinct from the runtime serve loop.** On first boot (no keystore on disk) the enclave opens a bootstrap listener on a dedicated port (`TWOD_HSM_PROVISIONING_VSOCK_*` env), accepts ONE provisioning connection, mints+seals, and **shuts the bootstrap listener down**. The runtime 0x40 serve loop (`serve_agent_loop`, separate port `TWOD_HSM_AGENT_VSOCK_*`) does NOT gain a provisioning path — this is a hard structural invariant with a regression test. Reuses `bind_vsock_listener` infra. | Mixing provisioning (one-shot, high-trust, operator-CA) and runtime (ongoing, any valid cap-bearer) in one listener creates an escalation surface: any vsock caller could attempt to re-provision. Different port + lifecycle = different trust boundary by construction. |
| **Q6** | **Agent keystore only.** G3 installs one `pq-agent-keystore-v1` body (agent secp256k1 keys + faucet state + counters). The producer ML-DSA keystore (long-term Block Producer key, 0x19 surface, TASK-1.x / TASK-5) is a **separate task** — different protocol, different threat model (long-term network key vs faucet budget), different lifecycle (one-shot at network deploy vs operational re-provisioning). The two keystores are independently sealed under their own domain-separated keys (TASK-7.2 AC#16). | Mixing both in one channel multiplies the review surface without security gain; each deserves its own analysis. TASK-18 / G3 is definitionally about agent caps + `enclave_scope_id`, which the producer surface does not have. |
| **Q7** | **Measurement allowlist pinned in the provisioner binary (MVP); on-chain `MeasurementRegistry` (TASK-1.4) is the documented future upgrade.** The provisioner verifies the enclave's SNP report measurement against an allowlist compiled into the provisioner binary at build. Allowlist = a **list** (rolling updates, multi-env), NOT a singleton. Hard rule: **production allowlist never includes lab/debug measurements**, and allowlists are environment-scoped (testnet/mainnet/lab never mix). | A reproducible Nix build ⇒ deterministic measurement ⇒ the pinned value is safe to compile in and auditable. A runtime config-file allowlist (b) is the attack surface we are closing (host swaps the file ⇒ provisioner trusts a malicious measurement). On-chain (c) is the correct end-state but TASK-1.4 is still To Do and cross-repo; making G3 depend on it would block un-gate. |
| **Q8** | **Re-provisioning is hard-rejected (`AlreadyProvisioned`).** Once a keystore is sealed and persisted, any further provisioning connection fails closed. There is NO recovery-tier "factory reset" path. A corrupt or missing sealed blob ⇒ boot fail-closed (NOT a re-provision path); the operator decommissions the enclave and deploys a new one (fund-loss over budget-multiplication). Disaster recovery is RESTORE_BACKUP (TASK-24, which carries counters forward), not re-provisioning. | A re-provision path is a factory-reset, and a factory-reset is death for the anti-replay model: the host drops the blob, re-provisions, and mints a second treasury key ⇒ budget multiplication (the AC#12 hole, just via re-provisioning instead of fleet-scope). Even recovery-quorum-gated (b) creates a legitimate path the host could try to trigger. Hard reject removes the path entirely. |

## §1 Threat model + trust boundaries

**Trust boundary:** the SEV-SNP enclave is trusted; the host (hypervisor / operator runtime / disk /
snapshots / vsock) is untrusted EXCEPT for the operator-CA keyholder(s), who are trusted to
provision a genuine enclave but NOT trusted to be inside the TEE (a compromised operator can still
select a malicious `admin_authority_pk` — that is operator-trust by design, §5).

**The adversary G3 must defeat:**

1. **Clone-with-copied-id (the 18-2 adversary).** A host that clones an enclave AND can select or
   copy `enclave_scope_id` defeats the 18-2 byte-compare (clone B carries A's id ⇒ the verifier
   accepts A's caps on B). G3 closes this by making the id **enclave-minted and host-uncontrollable**
   (Q3 + Q4 structural): the host never sees the plaintext id and the wire protocol does not carry
   it.

2. **MITM-config-substitution (the Q1 mutual direction).** After a genuine enclave attests, a host
   controlling vsock could substitute the config map in transit (point the enclave at a malicious
   authority, a different chain, etc.). G3 closes this with mutual auth: the config map is delivered
   inside the operator-signed boot-authorization envelope, whose nonce is channel-bound to the
   attestation `report_data` (Q1).

3. **Re-provisioning budget multiplication (the Q8 adversary).** A host that can trigger
   re-provisioning can factory-reset the keystore and mint a second treasury key. G3 closes this by
   hard-rejecting re-provisioning (Q8).

**Out of scope for G3** (closed elsewhere, documented here for completeness):

- **Host-rollback of the sealed blob** (checkpoints an old blob, replays it) — closed by the
  TASK-7.7 anchor anti-rollback mechanism (Option A epoch-lease), NOT by G3.
- **Honest active-active clones** (two live enclaves of one treasury key) — the §4 active-active
  prohibition of `agent-gateway-anti-rollback.md`; needs Option B's global ledger, deferred to
  TASK-20.
- **Compromised operator** (selects malicious `admin_authority_pk`) — operator trust by design.
- **`fleet_scope_id` collusion across clones** (provisioner gives two enclaves the same fleet id) —
  legitimate (fleet clones share it by design); closed for budget-sensitive ops by the 18-5
  `financial ⇒ scope_class==0` handler policy, not by G3.

## §2 The provisioning ceremony (happy path)

1. **Operator prepares.** Off-line, the operator generates the agent-keystore config (chain id,
   environment id, admin/recovery authority keypairs, backup-recovery ML-KEM keypair, fleet id,
   anchor root) and signs a boot-authorization envelope binding that config + the target measurement
   + a fresh ceremony nonce, under the operator CA (Q1, quorum = 1 key MVP).

2. **Enclave first boot (no keystore on disk).** The enclave opens the bootstrap vsock listener
   (Q5, dedicated port) and waits. The runtime serve loop is NOT yet started.

3. **Provisioner connects + mutual attestation (Q1).**
   - The provisioner opens a vsock connection to the bootstrap port and sends a challenge nonce `N_p`.
   - The enclave mints its own nonce `N_e`, emits an SNP attestation report whose `report_data` binds
     `(N_p, N_e, measurement, TCB)` (the channel-binding from Q1 — the report is unusable on any
     other session).
   - The provisioner verifies the report: VCEK/ASK/ARK chain (TASK-1.2 `snp_verify.rs`), measurement
     in the pinned allowlist (Q7), TCB not revoked, `N_p` echoed (channel-binding).

4. **Provisioner delivers config (Q2 + Q4).** The provisioner sends the operator-signed
   boot-authorization envelope over the same vsock session: the config map (8 public fields, basket
   B) + signature. The envelope's nonce = `N_e` (so it cannot be replayed on a different session).

5. **Enclave verifies + mints + seals (Q2 + Q3).** The enclave verifies the boot-auth signature vs
   the pinned operator CA, verifies the nonce, then:
   - mints `enclave_scope_id = getrandom(32)` in-TEE (basket A, the one host-uncontrollable field);
   - constructs `KeystoreBody { config: {…basket B…, enclave_scope_id, authority_epoch=0}, entries:
     [], counters: [], faucet: zero-state, audit: empty, freshness_epoch: 0, structural_version: 1 }`
     (genesis-equivalent, per Q2);
   - calls the existing `seal_body()` under the measurement-derived provisioning root.

6. **Persist + hand off.** The enclave returns the sealed blob to the host over the vsock session;
   the host writes it to the keystore path on disk. The enclave **shuts down the bootstrap listener**
   (Q5) and starts the runtime 0x40 serve loop on its dedicated port.

7. **Subsequent boots.** The bootstrap listener does NOT open (keystore present on disk). The enclave
   boots via the existing `unseal_agent_keystore_at_boot` path and starts the runtime serve loop
   directly. Any connection to a bootstrap port that is not listening is simply refused.

## §3 The `enclave_scope_id` provenance contract (AC#3 / AC#4 — the load-bearing AC)

This is the AC that makes 18-2's byte-compare enforceable in production. Stated as testable
invariants:

- **I1 — mint source.** `enclave_scope_id` is the output of `getrandom(32)` called INSIDE the
  enclave, at provisioning, exactly once. There is no code path that reads it from the provisioner
  config map, from a host file, or from any channel the host can influence.
- **I2 — wire absence (Q4 structural).** The provisioner-config map wire format (frozen in 25-2)
  has NO field for `enclave_scope_id`. A host cannot supply it because the protocol does not carry
  it. (Contrast: a runtime "reject if present" check would be weaker — a future field addition could
  silently relax it.)
- **I3 — persistence.** After mint, the id lives only inside the sealed `KeystoreConfig`, authenticated
  by the keystore AEAD. It is read back at every boot; it is never re-derived.
- **I4 — test-fixture exclusion (AC#4).** The `[0xe1;32]` / `[0xf1;32]` sentinels in `genesis_body()`
  / `reference_keystore_body()` are TEST FIXTURES behind `#[cfg(test)]` / lab feature gates. A
  release build MUST NOT source them. Acceptance test (25-3): a release/provisioning-path test
  proving the production code mints a fresh random id AND that a host-supplied id (if one were
  somehow injected) is not honored.
- **I5 — restore exception (AC#6, coordinate TASK-24).** `enclave_scope_id` is EXCLUDED from the
  RESTORE_BACKUP payload (it is enclave-local, like `anchor_root`). A restored keystore mints a NEW
  id in-TEE; caps minted before the backup fail the 18-2 compare post-restore (intended — restore =
  new identity). Operator-visible audit/status note required so the post-restore `0x43` on old caps
  is diagnosable.

**NB (the central caveat, repeated):** until these invariants land (25-3 + 25-4), the 18-2
byte-compare is correct code under a preview gate but is **security theater** in any build that
sources the sentinel. The TASK-18 un-gate slices (18-6..9) are hard-blocked on this AC.

## §4 Re-provisioning, boot-fail-closed, and disaster recovery (Q8)

- **Re-provisioning is rejected.** Once a keystore is persisted, any provisioning connection returns
  `AlreadyProvisioned` and does not advance state. There is no recovery-tier factory-reset opcode.
  DR is RESTORE_BACKUP (TASK-24), which carries counters/spend forward — NOT a reset.
- **Corrupt / missing sealed blob ⇒ boot fail-closed.** `unseal_agent_keystore_at_boot` already
  fails closed on a bad blob. This is NOT a re-provision trigger — the enclave refuses to start. The
  operator decommissions the enclave and deploys a new one (the old treasury key's funds are a
  loss event; this is preferred over any path that could mint a second key). The TASK-7.7 anchor
  retains the high-water marks for audit.
- **Disaster recovery = RESTORE_BACKUP, not re-provision.** RESTORE_BACKUP (TASK-24) is the one
  mechanism that installs a keystore into a fresh enclave post-provisioning, and it is recovery-tier
  (quorum) + counter-seeded (AC#11/#12) + identity-changing (I5). G3's provisioning channel and
  TASK-24's restore channel are **two distinct install paths** with different trust models; they
  must not share an opcode or a listener.

## §5 Residuals (honest limitations)

- **Compromised operator CA.** An operator with the CA key can sign a boot-auth for a malicious
  config (e.g., their own `admin_authority_pk`). This is operator-trust by design — authorities are
  external trust roots. Mitigation is operational (CA key hygiene, quorum > 1 post-MVP), not
  enclave-enforced.
- **Measurement allowlist is build-pinned (Q7).** A new enclave release requires a new provisioner
  release with an updated allowlist. This is the cost of (a) over (b/c); accepted for MVP, path to
  on-chain via TASK-1.4.
- **Quorum = 1 MVP.** The operator-CA and recovery-authority models both ship with quorum = 1 (one
  higher-tier key) for MVP, mirroring the existing capability-tier precedent. Threshold/quorum > 1
  is a post-MVP hardening follow-up (tracked in TASK-25 AC notes), not a G3 blocker.
- **No producer keystore coverage (Q6).** G3 installs the agent keystore only; the producer ML-DSA
  keystore has its own provisioning path (TASK-1.1 / TASK-5). The two channels are independent; an
  enclave running both keystores accepts two independent provisioning connections on two ports.

## §6 Scope boundary: producer keystore is NOT G3 (Q6)

The Agent Gateway keystore (`pq-agent-keystore-v1`, secp256k1 keys + faucet state + capability
counters + `enclave_scope_id`) and the producer keystore (`pq-seal-v1`, the long-term ML-DSA Block
Producer key for AuthorizationTicket / hard-fork signing) are **independently sealed, independently
provisioned, and independently threat-modeled**. G3 covers ONLY the former.

- **Producer provisioning** is owned by TASK-1.1 (platform-derived PQ seal attestation trust root)
  and TASK-5 (production HSM guest / prod enclave platform seal / SNP). The staging runbook is
  `pq-seal-v1-provisioning-runbook.md`.
- **Do not merge the two channels.** A future "consolidate provisioning" cleanup MUST NOT fold the
  producer and agent install paths into one listener/opcode — they have different trust roots,
  different lifecycles (producer = one-shot at network deploy, agent = operational), and different
  blast radii (producer key = the whole 2D chain).

## §7 Slicing (implementation plan, post-design-approval)

Each slice is a separate PR with its own matrix (Full for protocol/provenance, Reduced for narrow
follow-ons), per the AGENTS.md decision rules. The 18-x pattern: design → impl → matrix → compact
→ fixes → re-verify.

- **25-1 (THIS DOC) — design.** Locked decisions Q1-Q8 + ceremony + provenance contract + residuals.
  Full Matrix on the design (the input that catches inversions like the 18-2 "underspecified and
  partly inverted" finding before any code).
- **25-2 — provisioning channel protocol spec + impl skeleton.** Byte-exact wire format for the
  boot-authorization envelope + the config map (basket B, 8 fields, NO `enclave_scope_id` field —
  I2), mutual-auth message order, idempotency framing. Golden vectors + negative tests. Full Matrix
  (new wire format ⇒ cap format change-class risk).
- **25-3 — `enclave_scope_id` in-TEE mint.** Replace the sentinel with a `getrandom` mint at
  provisioning; enforce I1/I2/I4 (test-fixture gating, structural wire absence, release-path test).
  Full Matrix (this is the AC#3 load-bearing AC).
- **25-4 — production agent-gateway nix profile.** The `enclave.nix` profile that builds the
  `twod-hsm-agent-gateway` bin in release with `TWOD_HSM_STRICT_RELEASE_GUARDS=1` and the
  provisioning channel wired (NO `lab-agent-keystore-from-file`, NO `agent-*-preview` features
  until each is un-gated by its own 18-6..9 slice). Reduced Matrix.
- **25-5 — restore identity-change (coordinate TASK-24).** Wire I5: `enclave_scope_id` excluded
  from restore payload, fresh mint on restore, operator-visible audit note. Co-dependency with
  TASK-24.
- **25-6 — operator runbook.** Provisioning ceremony, measurement-allowlist management, TCB
  revocation handling, re-provisioning-is-rejected operator guidance. Doc-only, Reduced Matrix.

**Un-gate (TASK-18 18-6..9) stays hard-blocked on 25-3 + 25-4 landing.** Each per-op un-gate
remains its own irreversible production step with its own Full Matrix + human gate, per AGENTS.md.
