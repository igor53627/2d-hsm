# Agent Gateway provisioning channel — attested keystore install + in-TEE scope-id provenance (TASK-25 / G3)

The production deployment path for the Agent Gateway keystore. Today **no production
agent-gateway image exists** — every build is a debug build that loads a keystore from a host file
via the `lab-agent-keystore-from-file` feature, and `enclave_scope_id` is a TEST-ONLY sentinel
(`[0xe1;32]`). This doc specifies the **G3** gap carved out of TASK-18 (per the 2026-06-19 locked
maintainer decision Q6): a production release image, an attested AF_VSOCK + SNP channel that
installs the sealed keystore into the enclave, and an in-TEE-minted random `enclave_scope_id` so
the 18-2 scope-binding byte-compare is not security theater.

**Status: design (25-1, Understand phase).** Q1-Q8 below are the locked maintainer decisions
(2026-06-19). This doc is the input to a Full Matrix on the design before any implementation slice
(25-2 onward). It is NOT a wire-format spec yet — §N refers forward to the 25-2 protocol slice that
freezes the byte-exact framing.

**Scope: agent keystore ONLY.** The producer ML-DSA keystore (long-term Block Producer key, the
0x19 vsock surface, TASK-1.x / TASK-5) is a **separate task with its own lifecycle and threat
model** (see §6). G3 installs one `pq-agent-keystore-v1` body and mints one `enclave_scope_id`.

## §1 Threat model + trust boundaries

**Trust boundary:** the SEV-SNP enclave is trusted; the host (hypervisor / operator runtime / disk /
snapshots / vsock) is untrusted EXCEPT for the operator-CA keyholder(s), who are trusted to
provision a genuine enclave but NOT trusted to be inside the TEE (a compromised operator can still
select a malicious `admin_authority_pk` — that is operator-trust by design, §5).

**The adversary G3 must defeat:**

1. **Fresh-provision clone-with-copied-id (the 18-2 adversary, NARROWED).** A host that
   fresh-provisions a clone B (own sealed keystore, empty counters) AND replays a cap minted for
   enclave A against B's empty counter row (`highest == 0 ⇒ incoming == 1` accepted) — re-minting a
   second treasury key from one logical authority. G3 closes THIS by making `enclave_scope_id`
   **enclave-minted and host-uncontrollable** (Q3 + Q4 structural): the cap's signed `scope_identity`
   carries A's id, B's sealed id differs, the 18-2 byte-compare rejects. **NB — what G3 does NOT close
   (HIGH#3 from the 25-1 matrix, honest residual):** a host that copies the WHOLE sealed blob from A
   to a same-measurement clone B carries A's `enclave_scope_id` + counters + keys + faucet state
   verbatim; clone B unseals it under the same measurement-derived key and the 18-2 byte-compare
   PASSES (same id). The whole-blob clone is closed by the **TASK-7.7 anchor** (B's local counters
   are behind the anchor's high-water ⇒ adopt-forward / fail-closed) plus the **§4 active-active
   prohibition** (operator-procedural under Option A, Option B's ledger for enforced). G3 does not
   add a third mechanism; the un-gate (18-6..9) is hard-linked to TASK-7.7 anchor readiness, not
   only to 25-3/25-4.

2. **MITM-config-substitution (the Q1 mutual direction).** After a genuine enclave attests, a host
   controlling vsock could substitute the config map in transit. G3 closes this with a
   **two-round-trip channel binding (HIGH#1 fix)**: the enclave first emits its SNP report carrying
   `N_e` (its session nonce); the operator-side signature is then produced OVER the live transcript
   `(N_p, N_e, report_hash, config)` so it is unforgeable for any other session. An offline-only
   envelope cannot carry `N_e` (the offline signer does not know it); the design therefore uses a
   round-trip (enclave emits quote → operator/provisioner signs over `N_e` → returns) OR an online
   provisioner key (certified by the offline operator CA) that signs the live transcript. See §2.

3. **Re-provisioning for budget multiplication (the Q8 adversary, NARROWED + HONEST).** A host that
   can trigger re-provisioning can factory-reset the keystore. G3 hard-rejects an explicit
   re-provision opcode, **BUT a host that controls disk can delete the sealed blob and reboot**,
   making the enclave indistinguishable-from-first-boot (SEV-SNP has no host-independent NVRAM);
   `AlreadyProvisioned` cannot be enforced from blob presence alone (HIGH#2). The actual safety
   property is NOT "the enclave detects deletion" — it is that a re-provisioned enclave mints a NEW
   `enclave_scope_id` + a zero-spend keystore AND comes up strictly behind the TASK-7.7 anchor's
   persisted high-water marks, so the anchor's adopt-forward / fail-closed reconciles it (a clone
   with fresh counters against an anchor that has seen higher counters is rejected or adopted
   forward, never silently double-budgeted). Q8 is reformulated below in those terms.

**Out of scope for G3** (closed elsewhere, documented here for completeness):

- **Host-rollback of the sealed blob** AND **whole-blob clone** AND **deleted-blob re-provision** —
  all closed by the TASK-7.7 anchor anti-rollback mechanism (Option A epoch-lease), NOT by G3.
  The un-gate (18-6..9) is hard-blocked on TASK-7.7 anchor readiness in addition to 25-3/25-4.
- **Honest active-active clones** (two live enclaves of one treasury key) — the §4 active-active
  prohibition of `agent-gateway-anti-rollback.md`; needs Option B's global ledger, deferred to
  TASK-20.
- **Compromised operator** (selects malicious `admin_authority_pk`) — operator trust by design.
- **`fleet_scope_id` collusion across clones** (provisioner gives two enclaves the same fleet id) —
  legitimate (fleet clones share it by design); closed for budget-sensitive ops by the 18-5
  `financial ⇒ scope_class==0` handler policy, not by G3.

## §2 The provisioning ceremony (happy path)

1. **Operator prepares (offline).** The operator generates the agent-keystore config (chain id,
   environment id, admin/recovery authority **public** keys, backup-recovery ML-KEM **public** key,
   fleet id, anchor root) and stages it. **No signature yet** — the signature is produced in step 4b
   over the live transcript (HIGH#1 fix: an offline-only envelope cannot carry `N_e`).

2. **Enclave first boot (no keystore on disk).** The enclave opens the bootstrap vsock listener
   (Q5, dedicated port) and waits. The runtime serve loop is NOT yet started.

3. **Round 1 — provisioner challenge + enclave attestation.**
   - The provisioner opens a vsock connection to the bootstrap port and sends a challenge nonce `N_p`.
   - The enclave mints its own nonce `N_e`, emits an SNP attestation report whose `report_data = H(N_p ‖ N_e)`
     (measurement/TCB are bound by the VCEK signature over the whole report, NOT re-hashed into
     `report_data` — 25-1-rev Med fix: avoids a dummy-report round-trip to discover them).
   - The provisioner verifies the report: VCEK/ASK/ARK chain (TASK-1.2 `snp_verify.rs`), measurement
     in the pinned allowlist (Q7), TCB not revoked, `N_p` echoed.

4. **Round 2 — online provisioner key signs the live transcript (HIGH#1 fix; MVP realization).**
   - (4a) The provisioner (holding an online provisioner key whose cert chains to the offline
     operator CA) has the full verified transcript `(N_p, N_e, report_hash)` in hand from step 3.
   - (4b) The provisioner signs `Sig_PROV = Sign_PROV(CANONICAL(config, N_p, N_e, report_hash))` and
     sends `(config, Sig_PROV, provisioner_cert)` to the enclave over the same vsock session.
   - (4c) The enclave verifies the provisioner cert chain to the pinned operator CA, re-derives
     `(N_p, N_e, report_hash)` from its own session state, and verifies `Sig_PROV` over it.

   Because `Sig_PROV` covers `N_e` (which the provisioner only has because the enclave emitted it in
   step 3), the signature is **unforgeable for any other session** — this is the channel binding Q1
   requires. Replay on a different enclave/session fails (different `N_e` ⇒ signature mismatch). The
   `config` itself is also covered, so a host cannot substitute it in transit.

   (Offline-operator-with-round-trip alternative: the provisioner forwards the full tuple
   `(N_p, N_e, report_hash, config)` to the offline operator, who signs and returns; the enclave then
   verifies a single operator-CA signature. This is documented but is NOT the frozen MVP wire format
   — 25-2a freezes the online-provisioner-key form.)

5. **Enclave verifies + mints + seals (Q2 + Q3).** The enclave verifies the `provisioner_cert`
   chain to the pinned operator CA, re-derives `(N_p, N_e, report_hash)` from its own session state,
   and verifies `Sig_PROV` over `CANONICAL(config, N_p, N_e, report_hash)` (the MVP realization from
   Q1; no `measurement`/`TCB` in the signed payload — they are VCEK-bound, not re-hashed), then:
   - mints `enclave_scope_id = getrandom(32)` in-TEE (basket A, the one host-uncontrollable field);
   - constructs the `KeystoreBody` via the **production genesis constructor** (the provisioned-body
     initializer — see §3 I4: today only `#[cfg(test)]` helpers exist (`empty_body()` in
     `agent_keygen.rs:166`, `genesis_body()` in `boot_agent_keystore.rs:294`); **25-3 MUST introduce
     a non-test production constructor** and MUST NOT source the sentinel from a test helper). The
     constructor's exact field set (incl. `strict_recovery_counter`, audit cursors,
     `structural_version=1`, `freshness_epoch` init per the constructor) is authoritative; this doc
     does not duplicate it to avoid drift. Faucet policy fields (`per_dispense_max_amount`,
     `max_gas_limit`, …) are **intentionally inert /
     zero at genesis** and are set post-provisioning by CONFIGURE_TREASURY (the only op that may raise
     spend authority) — they are NOT in any basket (25-1 Medium fix);
   - calls the existing `seal_body()` under the measurement-derived provisioning root.

6. **Persist + hand off.** The enclave returns the sealed blob to the host over the vsock session;
   the host writes it to the keystore path on disk. The enclave **shuts down the bootstrap listener**
   (Q5) and starts the runtime 0x40 serve loop on its dedicated port. **Atomicity / partial failure
   (25-1 Medium fix):** the enclave commits the seal before emitting the blob, so a vsock send
   failure leaves the sealed keystore **in volatile enclave session memory, NOT persisted** (SEV-SNP
   has no host-independent NVRAM — the same premise that anchors HIGH#2; 25-1-rev2 Med fix). On the
   next boot the enclave reboots into the no-keystore state and re-runs the ceremony (a FRESH
   `enclave_scope_id` is minted — harmless, no double-budget because counters are zero and the anchor
   has seen nothing). A ceremony is **successful iff the host has persisted the emitted blob** — the
   enclave-side one-shot listener slot is consumed only on a completed handoff (send + ack); a
   failed/abandoned handoff releases the slot so the host may retry.
   The host obtains the sealed blob from the ceremony session that reached step 6; a stalled bootstrap
   connection is subject to a bootstrap-listener timeout (25-2 freezes the value). A timed-out / abandoned
   round-1 session MUST release the one-shot listener slot (not consume it) so a stalled attacker
   cannot deny provisioning indefinitely (25-1-rev claude-code consideration).

7. **Subsequent boots (and the honest deleted-blob story, HIGH#2).** The bootstrap listener opens
   ONLY when no keystore is on disk. SEV-SNP has no host-independent NVRAM, so the enclave **cannot
   cryptographically distinguish "genuine first boot" from "host deleted the blob and rebooted"** —
   both look identical (no keystore). The safety property is therefore NOT "detect deletion" but:
   a re-provisioned enclave mints a NEW `enclave_scope_id` + a zero-spend keystore AND, because the
   TASK-7.7 anchor is an EXTERNAL durable monotonic service, the new enclave comes up strictly
   behind the anchor's persisted high-water marks ⇒ the anchor's adopt-forward / fail-closed logic
   reconciles it (never silently double-budgets). Q8 is enforced by the anchor, not by blob-presence.
   **Load-bearing premise (25-1-rev Med fix):** the anchor's high-water marks are keyed on the
   STABLE treasury identity `(twod_chain_id, environment_identifier)` (see `AnchorResponse` in
   `agent_anchor.rs:748` — NOT keyed on `enclave_scope_id`), so a re-provision with a FRESH scope id
   still maps to the SAME anchor row and reconciles against the prior marks. If the anchor were keyed
   on `enclave_scope_id`, a fresh-id re-provision would evade the high-water and re-open exactly the
   budget-multiplication the fix claims to close — so this keying is a hard dependency, not an
   assumption: the 25-3/25-4 slices MUST verify the anchor row mapping is scope-id-independent.
   Disaster recovery (genuine restore) is the separate RESTORE_BACKUP path (TASK-24, 25-5), which is
   NOT the bootstrap listener and is recovery-tier + counter-seeded.

## §3 The `enclave_scope_id` provenance contract (AC#3 / AC#4 — the load-bearing AC)

This is the AC that makes 18-2's byte-compare enforceable in production. Stated as testable
invariants:

- **I1 — mint source.** `enclave_scope_id` is the output of `getrandom(32)` called INSIDE the
  enclave, at provisioning, exactly once. There is no code path that reads it from the provisioner
  config map, from a host file, or from any channel the host can influence.
- **I2 — wire absence (Q4 structural).** The provisioner-config map wire format (frozen in 25-2)
  has NO field for `enclave_scope_id`. A host cannot supply it because the protocol does not carry
  it. (Contrast: a runtime "reject if present" check would be weaker — a future field addition could
  silently relax it.)
- **I3 — persistence.** After mint, the id lives only inside the sealed `KeystoreConfig`, authenticated
  by the keystore AEAD. It is read back at every boot; it is never re-derived.
- **I4 — test-fixture exclusion (AC#4).** The `[0xe1;32]` / `[0xf1;32]` sentinels in `genesis_body()`
  / `reference_keystore_body()` are TEST FIXTURES behind `#[cfg(test)]` / lab feature gates. A
  release build MUST NOT source them. Acceptance test (25-3): a release/provisioning-path test
  proving the production code mints a fresh random id AND that a host-supplied id (if one were
  somehow injected) is not honored.
- **I5 — restore exception (AC#6, coordinate TASK-24).** `enclave_scope_id` is EXCLUDED from the
  RESTORE_BACKUP payload (it is enclave-local, like `anchor_root`). A restored keystore mints a NEW
  id in-TEE; caps minted before the backup fail the 18-2 compare post-restore (intended — restore =
  new identity). Operator-visible audit/status note required so the post-restore `0x43` on old caps
  is diagnosable.

**NB (the central caveat, repeated):** until these invariants land (25-3 + 25-4), the 18-2
byte-compare is correct code under a preview gate but is **security theater** in any build that
sources the sentinel. The TASK-18 un-gate slices (18-6..9) are hard-blocked on this AC.

## §4 Re-provisioning, boot-fail-closed, and disaster recovery (Q8 — HIGH#2 reformulation)

- **Re-provisioning is rejected AS AN OPCODE.** There is no recovery-tier factory-reset opcode on
  the bootstrap listener; a provisioned enclave does not accept a re-provision message.
- **The honest deleted-blob story (HIGH#2).** A host that controls disk can delete the sealed blob
  and reboot. SEV-SNP has no host-independent NVRAM, so the enclave cannot cryptographically
  distinguish "genuine first boot" from "host deleted the blob" — both look like "no keystore on
  disk" ⇒ the bootstrap listener opens. The safety property is therefore NOT "detect deletion"
  (impossible without a TEE NVRAM) but: the re-provisioned enclave mints a NEW `enclave_scope_id` +
  a zero-spend keystore, and because the TASK-7.7 anchor is an EXTERNAL durable monotonic service,
  the new enclave comes up strictly BEHIND the anchor's persisted high-water marks ⇒ adopt-forward /
  fail-closed reconciles it (never silently double-budgets). **Q8 is enforced by the anchor, not by
  blob-presence.** This is why the un-gate (18-6..9) is hard-blocked on TASK-7.7 anchor readiness,
  not only on 25-3/25-4.
- **Corrupt (but present) sealed blob ⇒ boot fail-closed.** `unseal_agent_keystore_at_boot` already
  fails closed on a bad-AEAD blob. The operator decommissions the enclave and deploys a new one.
- **Disaster recovery = RESTORE_BACKUP, not re-provision.** RESTORE_BACKUP (TASK-24) is the one
  mechanism that installs a keystore into a fresh enclave post-provisioning with CONTINUITY (it
  carries counters/spend forward from the backup, not zero), and it is recovery-tier (quorum) +
  counter-seeded (AC#11/#12) + identity-changing (I5). G3's provisioning channel and TASK-24's
  restore channel are **two distinct install paths** with different trust models; they must not
  share an opcode or a listener. **Hard gate (25-1 Medium fix):** TASK-24 RESTORE_BACKUP MUST NOT
  go live before 25-5 wires the identity-change (I5), or a restored keystore could carry the old
  `enclave_scope_id` forward and re-enable exactly the clone-replay G3 closes.

## §5 Residuals (honest limitations)

- **Compromised operator CA.** An operator with the CA key can sign a boot-auth for a malicious
  config (e.g., their own `admin_authority_pk`). This is operator-trust by design — authorities are
  external trust roots. Mitigation is operational (CA key hygiene, quorum > 1 post-MVP), not
  enclave-enforced.
- **Whole-blob clone is NOT closed by G3 (HIGH#3, explicit residual).** G3 closes fresh-provision
  clone (clone B with empty counters + A's cap → 18-2 reject). It does NOT close a host copying the
  WHOLE sealed blob to a same-measurement clone (same `enclave_scope_id`, same counters) — that is
  closed by the TASK-7.7 anchor (clone B's local counters are behind the anchor ⇒ adopt-forward /
  fail-closed) + the §4 active-active prohibition. The un-gate (18-6..9) is hard-blocked on TASK-7.7
  anchor readiness, not only on 25-3/25-4.
- **Deleted-blob re-provision is NOT closed by G3 (HIGH#2, explicit residual).** As above (§4) — the
  enclave cannot detect blob deletion; safety is anchor-mediated. Same un-gate blocker.
- **Measurement allowlist is build-pinned (Q7).** A new enclave release requires a new provisioner
  release with an updated allowlist. This is the cost of (a) over (b/c); accepted for MVP, path to
  on-chain via TASK-1.4.
- **Quorum = 1 MVP.** The operator-CA and recovery-authority models both ship with quorum = 1 (one
  higher-tier key) for MVP, mirroring the existing capability-tier precedent. Threshold/quorum > 1
  is a post-MVP hardening follow-up (tracked in TASK-25 AC notes), not a G3 blocker.
- **Online-provisioner-key verification surface (25-1-rev residual).** The MVP realization (Q1)
  uses an online provisioner key certified by the offline operator CA, so the enclave gains X.509-style
  cert-chain validation against the pinned CA — a materially larger and more error-prone verification
  surface than a single pinned-key Ed25519 verify. Accepted for the live-transcript binding it enables;
  the chain-validation impl is part of 25-2b (Full Matrix) and is a 25-3/-4 review focus.
- **No producer keystore coverage (Q6).** G3 installs the agent keystore only; the producer ML-DSA
  keystore has its own provisioning path (TASK-1.1 / TASK-5). The two channels are independent; an
  enclave running both keystores accepts two independent provisioning connections on two ports.

## §6 Scope boundary: producer keystore is NOT G3 (Q6)

The Agent Gateway keystore (`pq-agent-keystore-v1`, secp256k1 keys + faucet state + capability
counters + `enclave_scope_id`) and the producer keystore (`pq-seal-v1`, the long-term ML-DSA Block
Producer key for AuthorizationTicket / hard-fork signing) are **independently sealed, independently
provisioned, and independently threat-modeled**. G3 covers ONLY the former.

- **Producer provisioning** is owned by TASK-1.1 (platform-derived PQ seal attestation trust root)
  and TASK-5 (production HSM guest / prod enclave platform seal / SNP). The staging runbook is
  `pq-seal-v1-provisioning-runbook.md`.
- **Do not merge the two channels.** A future "consolidate provisioning" cleanup MUST NOT fold the
  producer and agent install paths into one listener/opcode — they have different trust roots,
  different lifecycles (producer = one-shot at network deploy, agent = operational), and different
  blast radii (producer key = the whole 2D chain).

## §7 Slicing (implementation plan, post-design-approval)

Each slice is a separate PR with its own matrix (Full for protocol/provenance, Reduced for narrow
follow-ons), per the AGENTS.md decision rules. The 18-x pattern: design → impl → matrix → compact
→ fixes → re-verify.

- **25-1 (THIS DOC) — design.** Locked decisions Q1-Q8 + ceremony + provenance contract + residuals.
  Full Matrix on the design (the input that catches inversions like the 18-2 "underspecified and
  partly inverted" finding before any code).
- **25-2a — provisioning channel protocol spec (byte-exact wire format).** Frozen wire format
  for the two-round-trip ceremony (Q1), the online-provisioner-key MVP: `provisioner_cert` (chains
  to the offline operator CA) + `Sig_PROV` over `CANONICAL(config, N_p, N_e, report_hash)`, and the
  config map (basket B = 7 operator-chosen public fields; basket C = enclave-init deterministic;
  NO `enclave_scope_id` field — I2) + golden vectors. **Reviewable INDEPENDENTLY of any code**
  (25-1 Medium fix: split the frozen format from the impl skeleton so the wire contract can be
  matrix-reviewed on its own). Full Matrix (new wire format ⇒ cap-format-change-class risk).
- **25-2b — provisioning channel impl skeleton.** The Rust impl of the 25-2a format: message
  order, idempotency framing, bootstrap-listener timeout, atomicity contract (§2 step 6), + the
  online-provisioner-key cert-chain verification. **Full Matrix** (25-1-rev Med fix: this slice
  introduces the provisioning state machine + the enclave-side cert-chain verification surface,
  which is more than narrow consumer code on a frozen format — the ordering/idempotency/timeout
  semantics + the chain-validation are themselves concurrency-sensitive security logic).
- **25-3 — `enclave_scope_id` in-TEE mint.** Replace the sentinel with a `getrandom` mint at
  provisioning; enforce I1/I2/I4 (test-fixture gating, structural wire absence, release-path test).
  **RNG-source constraint (25-1-rev Low fix):** the production TEE RNG path MUST use hardware
  entropy only (`RDRAND`/`RDSEED` or the SNP `MSG_KEY_REQ` entropy source) and MUST disable / exclude
  any host-controlled entropy source (e.g. `virtio-rng`) — otherwise a host influencing the entropy
  pool could dictate the minted id and bypass the 18-2 defense. Full Matrix (this is the AC#3
  load-bearing AC).
- **25-4 — production agent-gateway nix profile.** The `enclave.nix` profile that builds the
  `twod-hsm-agent-gateway` bin in release with `TWOD_HSM_STRICT_RELEASE_GUARDS=1` and the
  provisioning channel wired (NO `lab-agent-keystore-from-file`, NO `agent-*-preview` features
  until each is un-gated by its own 18-6..9 slice). Reduced Matrix.
- **25-5 — restore identity-change (HARD GATE on TASK-24).** Wire I5: `enclave_scope_id` excluded
  from restore payload, fresh mint on restore, operator-visible audit note. **Hard gate (25-1 Medium
  fix):** TASK-24 RESTORE_BACKUP MUST NOT go live before 25-5 — otherwise a restored keystore could
  carry the old id forward and re-enable clone-replay. Not just "coordinate" — a release ordering
  constraint.
- **25-6 — operator runbook.** Provisioning ceremony, measurement-allowlist management, TCB
  revocation handling, re-provisioning-is-anchor-mediated (not blob-presence) operator guidance.
  Doc-only, Reduced Matrix.

**Un-gate (TASK-18 18-6..9) stays hard-blocked on 25-3 + 25-4 + TASK-7.7 anchor readiness.** The
anchor dependency is the HIGH#2/HIGH#3 reformulation: G3 closes fresh-provision clone, but
whole-blob clone + deleted-blob re-provision are closed by the anchor, not by G3. Each per-op
un-gate remains its own irreversible production step with its own Full Matrix + human gate, per
AGENTS.md.
