### 6-4b residual (from the PR #74 compact, 2026-06-14)

The 6-4b branch compact (roborev job 7314) re-surfaced one **High-labelled** finding: the `0x45` anchor-commit REQUEST carries only `{scope, proposed state, nonce, request_id}` with **no enclave-authenticated MAC/signature or boot-attested session binding**, so a compromised host/relay can forge/substitute well-formed commit requests and make the anchor durably advance/sign attacker-chosen state.

**Adversarially verified DoS-only (2 independent agents) — NOT a wrong-accept / rollback, NOT a 6-4b regression, accepted residual:**
- **No wrong-accept:** the enclave seals/swaps/emits ONLY after `verify_commit_ack_bytes` passes — Ed25519 `verify_strict` against the **sealed** `anchor_root` + echo of the enclave's OWN fresh per-op CSPRNG nonce (`draw_commit_nonce`) + `{epoch, structural, marks, request_id}`. A host holds only the public `anchor_root` (verify-only) so it cannot mint any valid ACK; a replayed genuine ACK fails the fresh-nonce echo. The enclave never adopts host-proposed state.
- **No rollback / never wrong-accept on reconcile:** the anchor is forward-only (`advance_commit_epoch` checked, monotone); a forged commit can only ADVANCE the anchor's recorded high-water. On next-boot `reconcile` (per the codex-7323 + compact-7332 corrections): a **structural** advance — the GENERATE_KEYS case, which is exactly 6-4b's path — routes to `StructuralGap` → `FailClosed` (recoverable). An **epoch-only** advance with matching `structural_version` routes to `AdoptForward`, whose gate compares the re-hashed adopted marks to the **anchor-signed** `state.marks_digest` (NOT to local marks), plus the `marks_dominate_local` belt requiring the adopted marks to DOMINATE local. So marks that hash-mismatch the anchor-signed digest, or that don't dominate local, fail closed — but marks the host already made the anchor durably record (monotone, attacker-chosen), served with a matching raw-marks payload, CAN be adopted. Crucially that adoption only ever RAISES the enclave's counter/spend high-water (monotone up = the SAFE direction): inflated counter floors reject legitimate next-capabilities (counter-too-low) and inflated cumulative-spend exhausts faucet caps prematurely — a **DoS / cap-exhaustion fast-forward**, NEVER a wrong-accept. A wrong-accept (over-spend, or counter rollback-replay) would require LOWERING the high-water, which the forward-only anchor + the dominate belt forbid; `AnchorBehind` (a lower anchor) is not reachable by a forward-only host. LOWERING the anchor is an anchor-side capability, not the host's — the documented residual is "a compromised operator who can ALSO roll the anchor back" (`agent-gateway-anti-rollback.md §3/§7`).
- **DoS-only + already conceded:** the host already relays all enclave↔anchor traffic + controls VM scheduling, so it can already wedge custody by dropping commits — the explicitly **accepted** "liveness-DoS availability residual" (§7: "fail-closed — no fund loss, no rollback, and the host gains nothing"). Forging adds no new power.
- **Design property, not new:** "the request is not signature-bound" is a deliberate decision (§5, same model as the `0x41` boot request); the `0x45` request format was FROZEN in slice 6-1 (PR #70). 6-4b only makes the path REACHABLE — under the OFF-BY-DEFAULT, release-banned `agent-keygen-exec-preview` gate (non-production).

**Parked OPTIONAL hardening (defense-in-depth vs the host-DoS variant; NOT required for safety):** bind `0x45` commit requests to the boot-attested enclave session — e.g. an enclave-held ephemeral MAC/signing key derived during the attested boot handshake, which the anchor verifies before any durable commit/ACK. This is a `0x45` wire + **anchor-side** protocol change (6-1-level, and it requires the anchor implementation to enforce it), so it belongs with a future anchor-protocol hardening pass, not the preview-gated boot wiring. User decision (2026-06-14): accept the documented DoS-only residual, merge 6-4b, park this hardening here.

Source: PR #74 compact job 7314 + the 2-agent adversarial verification (DoS-only, no wrong-accept/rollback).

- **[altitude, Low] `encode_agent_response` fail-closed-arm fragility.** Two mutating opcodes (GENERATE_KEYS, SIGN_FAUCET_DISPENSE) each need a hand-written `=> encode_agent_error(SealFailed)` arm in the generic encoder so a mis-route can't emit an un-sealed mutation. The on-curve re-validation is **intentional defense-in-depth** (consistent with `public_identity_from_entry`), but it's redundant work on sealed-and-already-`validate()`d bytes. Mitigation when warranted: precompute a `HashSet<[u8;20]>` of transfer-key addresses (or cache per-entry addresses) once at keystore install/unseal so the scan is a set lookup. **Not a vulnerability** — the serve-port client is the untrusted SNP host which already controls VM scheduling (the accepted "serial single-client monopolization / availability-vs-host" non-goal, §8).
